# IMU 姿勢（猫背）アプリ — 実機計測スパイク（feasibility probe）

> 起票: Even G2 の IMU（3軸モーション）で「歩行時の猫背（頭の前傾）を検知 → 矯正リマインドする」アプリを作れるかの構想。
> 本 spec は **本実装ではなく、本実装に入る前に実機で前提を確定するためのスパイク（feasibility probe）の契約**を固定する。
> precedence: 本 spike spec（前提確定）→ 後続の本実装 spec（未起票）。

## 背景・なぜスパイクが先か

SDK 型定義（`@evenrealities/even_hub_sdk@0.0.10`）を直接確認した結果、IMU から取れるのは **`imuData = { x, y, z }` の3軸 `double` のみ**で確定している（コメント `EvenHub.pb.dart：x / y / z, double on wire`）。ジャイロ（角速度）単独・6/9軸フュージョン・クォータニオン等は **プロトコルレベルで存在しない**。

この前提だと、本命ロジックは「**加速度の重力成分から頭のピッチ角を出し、キャリブレーション基準との角度差で前傾を検知する**」になる（ジャイロ積分が無くてもピッチ/ロールはドリフトしないので相性は良い）。ただし以下が **机上では確定できず、ドキュメントにも無い**:

- `x/y/z` の**素性**（加速度なのか／単位・レンジ／静止時に重力 1G 相当のノルムが出るのか／前傾でどの軸がどう動くか）
- `ImuReportPace`（P100/P500/P1000）の**実レート・安定性**（取りこぼし有無）
- **バッテリー実消費**（`gpsnmeajp` メモでは P1000 でも 10分で10%。各 pace の実測が要る）
- **フィードバック手段**: グラスにスピーカー/振動は無い。アプリ実体はスマホ WebView なので **`navigator.vibrate()` / 音が WebView で効くか**が矯正 UX の生命線（要実機確認）
- **権限**: `app.json` の `permissions` 一覧（`network`/`location`/`g2-microphone`/`phone-microphone`/`album`/`camera`）に **IMU は無い → 無権限で取れる見込み**だが未確認
- **装着連動**（非装着時に IMU が出るか／`isWearing` と併用すべきか）
- **シミュレーター非対応**（実機のみ）の再確認

本実装をいきなり作ると、これらの不確実性で設計（使う pace・閾値・フィードバック方式）が決められない。よって **生値可視化スパイクで前提を埋めるのが先**。

## スコープ

**やること（スパイク）**:

- IMU の生値を実機で観測・記録する最小アプリ（新規 `apps/<name>`、既存アプリには手を入れない）。
- **コンパニオン画面（スマホ WebView の DOM）を主計測 UI** にする。グラス表示は背面で console が読めないため、生値・派生値を**スマホ側の通常 DOM** に出すのが現実的（数値・簡易グラフ・ログ表・CSV/JSON コピー）。
- グラス表示には要約 1〜2 行のみ（`{x,y,z}` 丸め値、推定ピッチ角、状態）。
- 操作:
  - 計測 開始/停止（`imuControl(true/false)`）
  - `pace` 切替（P100 / P500 / P1000）
  - **キャリブレーション**（「良い姿勢で静止」ボタンで基準ベクトル `g_ref` を保存し、以降 `g_ref` と現在ベクトルの角度差 θ をリアルタイム表示 → 猫背検知が成立しそうか目視確認）
  - **振動テスト**ボタン（`navigator.vibrate()` が効くか単体確認）
- ログ持ち帰り: サンプルを `bridge.setLocalStorage` かコンパニオン画面のバッファに蓄積し、後でコピーできる形にする（背面 console に頼らない）。

**観測軸（このスパイクで埋める matrix）**:

| # | 観測項目 | 取り方 | 埋める値 |
|---|---|---|---|
| 1 | `x/y/z` の素性・レンジ | 静止水平で値を見る／頭を前後左右に倒す | 単位・静止ノルム・前傾でどの軸が動くか |
| 2 | 実レート | 受信タイムスタンプ間隔を pace 別に計測 | 各 pace の実 Hz・取りこぼし有無 |
| 3 | ピッチ角の妥当性 | `g_ref` 基準の θ を既知角度（例 0/15/30°うつむき）で対照 | θ が体感前傾と一致するか |
| 4 | 歩行ノイズ耐性 | 歩きながら θ を移動平均（1〜2s）で平滑 | 平滑後 θ が安定するか・必要な窓長 |
| 5 | 振動フィードバック | `navigator.vibrate([...])` を実機で叩く | OS/機種で鳴る/鳴らないか |
| 6 | バッテリー消費 | 各 pace で N 分連続計測し `batteryLevel` 差分 | %/10分の実測 |
| 7 | 権限・装着 | 無権限で IMU 出るか／非装着時の挙動 | app.json に IMU 権限要否・装着連動 |
| 8 | シミュレーター | simulator で IMU が来るか | 実機のみか再確認 |

**やらないこと（本実装ではない）**:

- 本番の猫背判定ロジックの作り込み（状態機械・通知の抑制・履歴/統計・目標設定）。
- 矯正 UX の磨き込み（警告デザイン・通知頻度チューニング・ゲーミフィケーション）。
- 永続的な常時バックグラウンド計測の設計（電池実測の結果を見てから）。
- ネットワーク送信・サーバ連携（スパイクはオンデバイス完結、egress を増やさない）。

## 設計契約（スパイク）

- **隔離**: 新規 `apps/<name>`（命名は実装着手時に確定。例 `nekoze` / `posture-spike`）。既存 `g2hermes` / `hisho` / `ui-lab` には触れない。雛形は everything-evenhub の `quickstart` / `template` スキルに従う。
- **構成**: 既存アプリ同様 Vite + TS + `@evenrealities/even_hub_sdk`。`bridge` 経由で `imuControl` と `onEvenHubEvent`（`sysEvent.eventType === IMU_DATA_REPORT` → `sysEvent.imuData`）。
- **純粋関数の単体テスト対象**: ベクトル正規化・角度差 θ 算出・移動平均・レート集計・CSV 整形は純関数として切り出しユニットテスト。実 I/O（`imuControl`/`vibrate`/`onEvenHubEvent`）は device-io として skip。
- **権限**: まず `permissions: []`（または `network` 無し）で IMU が出るか確認。出なければ要権限としてドキュメントに反映。
- **セキュリティ**: 新たな秘密情報・ネットワーク egress なし。計測ログは端末内（localStorage / 画面）に留め、外部送信しない。
- **配布**: 実機検証は QR サイドロード（ホットリロード）優先。GPS のように「サイドロードで権限拒否、Hub アップロード後のみ動く」罠が IMU にもあるかは #7 で確認。

## 完了の定義（スパイクとして）

上記 matrix（#1〜#8）が実測値で埋まり、本実装の判断材料が揃うこと。具体的には:

- **go/no-go**: `x/y/z` から前傾ピッチ角が安定して取れ、歩行ノイズを平滑で除けることが確認できるか（＝猫背検知が成立するか）。
- **設計パラメータの確定**: 本実装で使う `pace`（電池と精度のトレードオフ）、移動平均の窓長、θ の警告閾値の目安。
- **フィードバック方式の確定**: スマホ振動が使えるか／グラス表示のみに割り切るか。
- **制約の明文化**: 電池の現実的な連続稼働時間、必要権限、装着連動、シミュレーター可否。

これらが揃った時点で、本実装 spec（猫背矯正アプリ本体）を別途起票する。

## 既知の限界（本実装まで持ち越す前提）

- **頭の角度 ≠ 背骨の姿勢**。メガネは頭しか測れず、首が前に出るタイプの猫背には強いが、頭は水平のまま背中だけ丸める代償姿勢は検知できない。あくまで相関プロキシであることを本実装の UX 説明にも明記する。
- 電池制約により**常時監視は非現実的**な可能性が高い → 「散歩セッション中だけ ON」等の運用前提になる見込み。

## 実装状況（2026-06-13）

スパイクアプリ **`apps/imu-lab`** を実装・push 済み（ブランチ `claude/even-g2-gyro-sensor-5qhb0k`）。机上検証（`bun test` 30件 / `tsc -b` / `vite build` / `biome check`）はすべて通過。**残りは実機での計測のみ**。

実装の要点:
- ロジックは純関数に分離（`src/imu/math.ts`=ベクトル/角度/レート/CSV、`state.ts`=共有状態+θ算出、`reducer.ts`=計測状態遷移、`glass.ts`=グラス行生成）。ユニットテスト有。
- `src/imu/useImu.ts` がブリッジ配線（`imuControl` / `onEvenHubEvent` の `IMU_DATA_REPORT` 購読 / `getDeviceInfo` で電池 / `navigator.vibrate`）。SDK 未注入環境では `connected=false` で no-op。
- 主計測 UI = `src/companion/ImuLab.tsx`（スマホ DOM）。グラスは `src/glass/`（even-toolkit `useGlasses`）で副表示。
- θ（猫背指標）= キャリブ基準ベクトルと平滑後の現在ベクトルのなす角。軸割り当てが未知でも成立。歩行ノイズは 1.5s 移動平均で除去。
- `app.json` は `permissions: []`（IMU が無権限で取れるかを #7 で確認する狙い）。

## 次にやること（実機計測 — ここから）

> リモート作業の引き継ぎメモ。手元の PC（`.env` 等は不要、ネットワーク送信なし）で実施する。

### 手順

1. ルートで `bun install`（済みなら不要）。
2. `cd apps/imu-lab && bun run dev`（`:5175`、`--host` 付き）。
3. `evenhub qr --url "http://<PCのIP>:5175"` で実機サイドロード（ホットリロード可）。
4. グラス装着 → スマホ画面で「計測開始」。
5. **良い姿勢で静止して「キャリブ」** → 基準ベクトルを記録。
6. うつむく / 歩く / 頭を前後左右に倒す、で観察。
7. pace を P100 / P500 / P1000 と変え、レートと電池を比較。
8. 「CSV コピー」でログを持ち帰り分析。

### 埋める計測チェックリスト（spec 上部 matrix #1〜#8 に対応）

- [ ] **#1 素性**: 静止水平で `|v|`（ノルム）が一定値になるか＝加速度か。その値（重力 1G 相当が何になるか＝単位）。前傾でどの軸（x/y/z）が主に動くか。
- [ ] **#2 実レート**: P100 / P500 / P1000 それぞれの「実効 Hz」表示と取りこぼし感。
- [ ] **#3 θ の妥当性**: 既知の前傾（0° / 15° / 30° うつむき）で θ が体感と一致するか。
- [ ] **#4 歩行ノイズ耐性**: 歩きながら「平滑(1.5s)」値と θ が安定するか。窓長は足りるか。
- [ ] **#5 振動**: 「振動テスト」ボタンの結果（true/false）。実機・OS・機種をメモ。
- [ ] **#6 電池**: 各 pace で N 分計測し「消費(開始比)」を記録（%/分）。
- [ ] **#7 権限/装着**: `permissions: []` のまま IMU が出たか。QR サイドロードで出たか（GPS は出ない罠あり→ Hub アップロード後のみの可能性）。非装着時にサンプルが来るか。
- [ ] **#8 シミュレーター**: simulator で IMU が来ないこと（実機のみ）を再確認。

> 計測結果は本ファイルにこの見出しの下で表として追記する（TTS プローブ spec の「実機プローブ結果」節と同じ運用）。

### 計測後の判断（go/no-go と本実装への分岐）

- **go なら**: 確定した設計パラメータ（使う pace・移動平均窓長・θ 警告閾値・フィードバック手段＝振動 or グラス表示のみ）を持って、**本実装 spec（猫背矯正アプリ本体・新規アプリ）を別途起票**する。`apps/imu-lab` の純関数（math/state/reducer）は本体へ流用可能。
- **no-go なら**: 「頭の 3 軸では猫背を実用的に検知できない」理由（θ が歩行で暴れる/電池が持たない/振動が効かない等）を本ファイルに記録して打ち切り、別アプローチ（R1 リング併用・条件付き起動など）を検討する。
