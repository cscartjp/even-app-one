# ディスプレイと UI 詳解

> 原文: https://github.com/nickustinov/even-g2-notes/blob/main/docs/display.md
> ※ 本ページはコミュニティ製リファレンス [even-g2-notes](https://github.com/nickustinov/even-g2-notes) の非公式な日本語要約です。原文も独自調査に基づく非公式情報です。
> グリフ調査はシミュレーター（`@evenrealities/evenhub-simulator` v0.0.7、2025年2月）でのテスト結果。実機では異なる可能性があります。

## キャンバス

576×288 ピクセル。原点 (0, 0) は左上、X 右方向、Y 下方向。緑の micro-LED で、すべての色はホストアプリ側で 4bit グレースケール（緑 16 階調）に変換されてから送信される。

## コンテナモデル

UI は **コンテナ**（キャンバス上に絶対座標で置く矩形領域）で構成する。CSS も flexbox も DOM もない。ピクセル座標でコンテナを定義し、グラスのファームウェアが描画する。

**ルール:**
- 1 ページにつき最大 **画像コンテナ 4 個 + その他コンテナ 8 個**（計 12、型の混在可）
- ちょうど **1 つ** のコンテナに `isEventCapture: 1` を設定する — これが入力イベントを受け取るコンテナ
- `containerTotalNum` は実際に渡すコンテナ数と一致させる
- コンテナは重ねられる（後に宣言したものが上に描画される）
- z-index 制御は宣言順以外にない

## 共通プロパティ

| プロパティ | 型 | 範囲 | 備考 |
|---|---|---|---|
| `xPosition` | number | 0–576 | 左端ピクセル |
| `yPosition` | number | 0–288 | 上端ピクセル |
| `width` | number | 0–576 | 画像は 20–288 |
| `height` | number | 0–288 | 画像は 20–144 |
| `containerID` | number | 任意 | ページ内で一意。更新時に使用 |
| `containerName` | string | 最大 16 文字 | ページ内で一意。更新時に使用 |
| `isEventCapture` | number | 0 / 1 | ちょうど 1 つだけ 1 にする |

## ボーダーと装飾

リスト・テキストコンテナで利用可（画像は不可）:

| プロパティ | 範囲 | 備考 |
|---|---|---|
| `borderWidth` | 0–5 | 0 = ボーダーなし |
| `borderColor` | 0–15（リスト）/ 0–16（テキスト） | 階調レベル。実用値: 5 = 控えめなグレー、13 = 明るめ |
| `borderRadius` | 0–10 | 角丸 |
| `paddingLength` | 0–32 | 全辺一律のパディング |

**背景色・塗りつぶしのプロパティは存在しない。** 視覚装飾はボーダーのみ。

## テキストコンテナ（`TextContainerProperty`)

主力コンテナ。プレーンテキストを左寄せ・上寄せで描画する。中央寄せ・右寄せ・フォントサイズ指定・太字/斜体/下線はいずれも不可。

**コンテンツ文字数の上限:**

| API | 上限 |
|---|---|
| `createStartUpPageContainer` | 1000 文字 |
| `textContainerUpgrade` | 2000 文字 |
| `rebuildPageContainer` | 1000 文字 |

**描画挙動:**
- テキストはコンテナ幅で折り返す。コンテンツがコンテナ高を超え、かつ `isEventCapture: 1` なら、ファームウェアが内部スクロールし、スワイプ/リングで読み進められる
- `SCROLL_TOP_EVENT` / `SCROLL_BOTTOM_EVENT` は内部スクロールが上端/下端に達したときだけ発火する**境界イベント**（生のジェスチャーイベントではない）
- 実際には多くのアプリが手動ページネーション（約 400 文字/ページ分割）を採用する。改ページの制御・進捗表示・`rebuildPageContainer` の 1000 文字制限回避のため
- `isEventCapture: 1` のないコンテナはスクロール入力を受けないので、あふれたテキストはクリップされる
- `\n` で改行できる
- Unicode 文字（`▲` `━` `─` や矢印など）が使える — プログレスバーやインジケーターに有用
- 全画面（576×288）のテキストコンテナで約 400–500 文字が目安（文字幅に依存）
- 「中央寄せ」したい場合はスペースで手動パディングするしかない

**`textContainerUpgrade` による部分更新:**
- ページ全体を再構築せずにテキストをインプレース更新する
- `containerID` と `containerName` の一致が必要
- `contentOffset` / `contentLength` で文字列の部分置換が可能
- 戻り値は `boolean`
- **注意:** シミュレーターでは部分更新でも画面全体が再描画される。実機ではより滑らか

## フォントと Unicode 対応

ファームウェア組み込みの単一 LVGL フォントのみ。フォント選択・サイズ指定は不可で、**等幅フォントではない**（文字ごとに幅が異なる）。フォントにない文字は**黙ってスキップされる**（豆腐 □ すら表示されない）。

> 💡 **日本語開発者向けの注意:** 原文で表示確認されている CJK 文字は一部の漢字（日火水木金土年月時分秒人大小上下左右前後東西南北 など）と注音符号・全角英数字のみ。**ひらがな・カタカナ・大半の漢字は表示できない可能性が高い**（未対応文字は無表示でスキップされる）。日本語 UI を作る場合は画像コンテナでの描画を検討すること。

### 全角文字による等幅ワークアラウンド

フォントがプロポーショナルなため、ASCII でグリッド配置（ゲーム盤面・整列レイアウト）を作ると崩れる。回避策は **CJK 全角文字**を使うこと。全角文字同士は Unicode 規格（East Asian Width `F`/`W`）により同一幅が保証される。

- **全角スペース** `　` — グリッド整列には半角スペースの代わりにこれを使う
- **全角英字** `Ａ`–`Ｚ`, `ａ`–`ｚ`（U+FF21–U+FF5A）
- **全角数字** `０`–`９`（U+FF10–U+FF19)
- **CJK 漢字**（日火水木金土 など）・**注音符号**（ㄌㄛㄜ など）

ASCII → 全角変換は文字コードに `0xFEE0` を足す:

```typescript
// 'A' (U+0041) → 'Ａ' (U+FF21)
function toFullwidth(str: string): string {
  return str.replace(/[\x20-\x7E]/g, (ch) =>
    ch === ' ' ? '　' : String.fromCharCode(ch.charCodeAt(0) + 0xFEE0)
  )
}
```

**ゲームでの実用 Tips:**
- 空のグリッドセルには半角スペースや `□` でなく `　`（全角スペース）を使う
- `▦` や `●` のような非全角文字は全角と完全同幅ではないが、実用上はほぼ揃う
- **スクロールバー対策:** グリッド最終行の後の `\n` が空行を作り、コンテナ高を超えてスクロールバーが出ることがある。末尾の改行を削除する

### アプリで使える組み合わせ

プログレスバー・UI:
- `━`（太）/ `─`（細）の横線
- `█▇▆▅▄▃▂▁` — 8 段階の縦方向フィル
- `▉▊▋▌▍▎▏` — 7 段階の横方向フィル
- `▒` — 空部分のフィル

ゲーム・図:
- `●○` / `■□` — 塗り/空
- `▲△▶▷▼▽◀◁` — 方向三角、`◢◣◤◥` — コーナー三角
- `─│┌┐└┘├┤┬┴┼` — 罫線、`╭╮╯╰` — 角丸
- `★☆` — 星、`♠♤ ♥♡ ♣♧` — カードスート

## リストコンテナ（`ListContainerProperty`)

ファームウェアがネイティブ描画するスクロール可能なリスト。スクロールによる選択ハイライト移動はグラス側が自動処理する — アプリ側で選択状態を追跡する必要はない。

**`ListItemContainerProperty` のフィールド:**

| プロパティ | 範囲 | 備考 |
|---|---|---|
| `itemCount` | 1–20 | `itemName.length` と一致させる |
| `itemWidth` | ピクセル | 各行の幅。`0` = 自動。通常は `コンテナ幅 - 2×padding` |
| `isItemSelectBorderEn` | 0 / 1 | 選択中アイテムのハイライトボーダー表示 |
| `itemName` | string[] | 各 64 文字まで。各アイテムのラベル |

**挙動と制限:**
- スクロールはファームウェアが処理。アプリの介入（`rebuildPageContainer`）不要
- `SCROLL_TOP_EVENT` / `SCROLL_BOTTOM_EVENT` はリストの上端/下端に達したときのみ発火（境界イベント）
- クリックイベントは `listEvent` 経由で `currentSelectItemIndex` / `currentSelectItemName` を報告
- アイテムごとのスタイル指定不可（色・アイコン・2 行目テキストなし）
- アイテム高は `コンテナ高 ÷ itemCount` で自動計算（指定不可）
- アイテムは 1 行プレーンテキストのみ
- アイテムのインプレース更新は不可 — 変更には `rebuildPageContainer` が必要
- アイテム間のセパレーター線なし（ボーダーはリスト全体のみ）

**注意（Gotcha）:** リストがあるとスクロール処理を乗っ取られる。スクロールイベントは `listEvent` として届き（`textEvent`/`sysEvent` ではない）、選択ハイライトはデバイスが自動で動かす。アプリはクリック/ダブルクリックに応答するだけでよい。

## 画像コンテナ（`ImageContainerProperty`)

グレースケール画像を表示する。すべての画像データはホストアプリ側で 4bit グレースケール（緑 16 階調）に変換されてから送信される。

| 制約 | 値 |
|---|---|
| 幅 | 20–288 px |
| 高さ | 20–144 px |
| 色 | ホスト側で 4bit グレースケール変換（`imageToGray4`） |
| データ形式 | `number[]` / `Uint8Array` / `ArrayBuffer` / base64 文字列 |
| 同時送信 | **不可** — 画像更新は逐次キューイングする |
| 起動フェーズ | `createStartUpPageContainer` 中は画像データを送れない — 空のプレースホルダを作り、後から `updateImageRawData` で更新 |
| タイリング | 画像データがコンテナ寸法より小さいとハードウェアが繰り返し表示する — 必ず画像サイズをコンテナに合わせる |

**`imageData` の形式:**
- **`number[]`（推奨）** — PNG ファイルのバイト列（`Array.from(new Uint8Array(pngBuffer))`）または生のグレースケールピクセル値
- **base64 文字列** — base64 PNG（`canvas.toDataURL('image/png')` から `data:image/png;base64,` プレフィックスを除去）
- **`Uint8Array` / `ArrayBuffer`** — SDK がシリアライズ時に `number[]` へ自動変換

**画像処理のアドバイス:**
- 色情報は捨てられるのでグレースケール画像を送る
- **アプリ側で 1bit ディザリングしないこと** — ホストの 4bit ダウンサンプリングのほうが高品質。手動の Floyd–Steinberg ディザはノイズ状の緑ドットになる
- タイリング防止のため画像寸法はコンテナ寸法に一致させる
- ベストプラクティス: アスペクト比を保ってリサイズ → 黒キャンバスに中央配置（黒 = micro-LED 消灯）→ グレースケール変換（BT.601 輝度: `0.299R + 0.587G + 0.114B`）→ PNG エンコード → `number[]` か base64 で送信
- 16 階調しかないため細部の多い画像は劣化する。シンプルに保つ
- グラスのメモリは限られる — 頻繁な画像更新は避ける
